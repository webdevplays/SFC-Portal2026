# Dokploy Deployment & MySQL Integration Guide - Saint Francis Clinic 🩺

This guide provides step-by-step instructions for deploying the Saint Francis Clinic management application on Dokploy (a modern, Docker-based self-hosted PaaS) and establishing a secure connection to a MySQL database container.

---

## 📅 Part 1: Setting up MySQL in Dokploy

1. **Log in to Dokploy**: Access your Dokploy administrator dashboard.
2. **Create a MySQL Database Service**:
   - Inside your Project, click on **Create Service** and choose **Database** -> **MySQL**.
   - Provide a service name (e.g., `sfclinic-mysql`).
   - Dokploy will automatically spin up a dedicated MySQL Docker container.
3. **Retrieve Credentials**:
   - Go to the **Environment** tab of the MySQL service.
   - Note the connection details generated by Dokploy:
     - **Database Name**: (e.g., `sfclinic`)
     - **Username**: (e.g., `root` or `sfclinic_user`)
     - **Password**: The auto-generated secure password.
     - **Internal Host**: `sfclinic-mysql` (If deploying on the same Dokploy Docker network)
     - **External Host**: Your VPS IP address (If connecting from outside the Dokploy network)
     - **Port**: `3306`

---

## 🗄️ Part 2: Importing the SQL Schema

1. **Access phpMyAdmin** or another database client connected to your Dokploy MySQL service. (Dokploy allows spawning a phpMyAdmin service on the same network).
2. Select your newly created database (`sfclinic`) from the left sidebar.
3. Click the **Import** tab on the top menu bar.
4. Click **Choose File** / **Browse** and upload the `mysql-schema.sql` (or `schema.sql`) from the root folder of this project.
5. Ensure the format is set to **SQL** and click **Import** or **Go** at the bottom.
6. The database tables, structural constraints (foreign keys), and initial seed data will be instantly created!

---

## ⚡ Part 3: Switching Node.js Backend to MySQL

To link this Node.js Express server to your MySQL database on Dokploy, set your connection parameters as Environment Variables in Dokploy:

1. In the **Environment Variables** tab of your Dokploy Application settings, add the following variables:
   ```env
   # MySQL Configuration Details
   DB_HOST=sfclinic-mysql  # Use Dokploy's internal service name for fast, direct Docker networking
   DB_USER=sfclinic_user
   DB_PASSWORD=your_secure_password
   DB_NAME=sfclinic
   DB_PORT=3306
   NODE_ENV=production
   ```
2. The application uses these variables to initialize a pool connection.
3. Alternatively, if no credentials are set, the application runs in stable fallback mode which persists data safely within a local JSON file (`data/db.json`).

---

## 🚀 Part 4: Deploying Node.js App on Dokploy

Dokploy supports Node.js applications natively via Nixpacks, Dockerfile, or pre-built images:

1. **Create an Application**:
   - Inside your Project, click on **Create Service** -> **Application**.
   - Select your Git provider (GitHub/GitLab) or deploy using a Git URL.
   - Point to this repository or branch.
2. **Configure Port**:
   - Set the port to **`3000`** in Dokploy's application settings.
3. **Configure Build Settings (Nixpacks)**:
   - Dokploy's default build tool is **Nixpacks**, which automatically detects Node.js, installs dependencies, runs `npm run build`, and starts the server using `npm start`.
   - Ensure you are running Node.js version `18.x` or higher.
4. **Deploy**:
   - Click **Deploy**. Dokploy will pull the code, compile the assets, compile the Node.js backend using `esbuild`, and spin up your container.
   - Once deployed, Dokploy will issue an automated SSL certificate and bind it to your chosen domain.

## 🛠️ Part 5: Troubleshooting & Common Build Errors

### ❌ Error 1: Database Connection Refused (`ECONNREFUSED`)

- **Why this happens:**
  The Node.js container cannot find the MySQL container at the specified `DB_HOST`.
- **How to Solve:**
  1. Ensure both the Application and the MySQL service are on the same **Dokploy Docker Network**.
  2. Use the exact **internal service host name** (e.g., `sfclinic-mysql`) as `DB_HOST`, not `localhost` or `127.0.0.1`.

### ❌ Error 2: Missing Foreign Key Schema Mismatch on Import

- **Why this happens:**
  Running structured schema imports while foreign keys are active can fail if table creation order is misaligned.
- **How to Solve:**
  Add `SET FOREIGN_KEY_CHECKS = 0;` at the very beginning of your SQL import execution, and restore it using `SET FOREIGN_KEY_CHECKS = 1;` at the end.
