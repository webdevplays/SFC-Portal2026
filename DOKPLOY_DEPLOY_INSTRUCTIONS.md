# Dokploy Deployment & PostgreSQL Integration Guide - Saint Francis Clinic 🩺

This guide provides step-by-step instructions for deploying the Saint Francis Clinic management application on Dokploy (a modern, Docker-based self-hosted PaaS) and establishing a secure connection to a PostgreSQL database container.

---

## 📅 Part 1: Setting up PostgreSQL in Dokploy

1. **Log in to Dokploy**: Access your Dokploy administrator dashboard.
2. **Create a PostgreSQL Database Service**:
   - Inside your Project, click on **Create Service** and choose **Database** -> **PostgreSQL**.
   - Provide a service name (e.g., `sfc-portal-sfcpostdb-snp1ir` or similar).
   - Dokploy will automatically spin up a dedicated PostgreSQL Docker container.
3. **Retrieve Credentials**:
   - Go to the **Environment** tab of the PostgreSQL service.
   - Note the connection details generated by Dokploy:
     - **Database URL (or Connection URL)**: (e.g., `postgresql://sfcuser:your_password@sfc-portal-sfcpostdb-snp1ir:5432/sfcdb`)
     - **Database Name**: (e.g., `sfcdb`)
     - **Username**: (e.g., `sfcuser`)
     - **Password**: The auto-generated secure password.
     - **Internal Host**: `sfc-portal-sfcpostdb-snp1ir` (If deploying on the same Dokploy Docker network)
     - **Port**: `5432`

---

## 🗄️ Part 2: Importing the SQL Schema

1. **Access your PostgreSQL Database client** (e.g. pgAdmin, DBeaver, or via Dokploy CLI terminal) connected to your Dokploy PostgreSQL service.
2. **Execute the SQL Schema Script**:
   - Run the contents of the `postgres-schema.sql` file from the root folder of this project in your database.
   - This will instantly create the required database tables, relational schemas, indexes, and initial configurations!

---

## ⚡ Part 3: Switching Node.js Backend to PostgreSQL

To link this Node.js Express server to your PostgreSQL database on Dokploy, set your connection parameters as Environment Variables in Dokploy:

1. Go to the **Environment Variables** tab of your Dokploy Application settings.
2. Add **ONE** of the following variables (Database URL is recommended):
   ```env
   # Recommended: Complete Connection URL
   DATABASE_URL=postgresql://sfcuser:your_password@sfc-portal-sfcpostdb-snp1ir:5432/sfcdb

   # OR configure individual variables:
   DB_HOST=sfc-portal-sfcpostdb-snp1ir
   DB_USER=sfcuser
   DB_PASSWORD=your_password
   DB_NAME=sfcdb
   DB_PORT=5432

   NODE_ENV=production
   ```
3. **CRITICAL STEP**: After updating environment variables on Dokploy, you **MUST click Redeploy or Restart** on the application container to load the new variables into Node.js.

---

## 🚀 Part 4: Deploying Node.js App on Dokploy

Dokploy supports Node.js applications natively via Nixpacks, Dockerfile, or pre-built images:

1. **Create an Application**:
   - Inside your Project, click on **Create Service** -> **Application**.
   - Select your Git provider (GitHub/GitLab) or deploy using a Git URL.
   - Point to this repository or branch.
2. **Configure Port**:
   - Set the port to **`3000`** in Dokploy's application settings.
3. **Configure Build Settings (Nixpacks)**:
   - Dokploy's default build tool is **Nixpacks**, which automatically detects Node.js, installs dependencies, runs `npm run build`, and starts the server using `npm start`.
   - Ensure you are running Node.js version `18.x` or higher.
4. **Deploy**:
   - Click **Deploy**. Dokploy will pull the code, compile the assets, compile the Node.js backend, and spin up your container.
   - Once deployed, Dokploy will issue an automated SSL certificate and bind it to your chosen domain.

## 🛠️ Part 5: Troubleshooting & Common Build Errors

### ❌ Error 0: Server returns an HTML page instead of JSON (Caddy Server Takeover)

- **Why this happens:**
  Since this is a full-stack Vite React application, there is an `index.html` file in the root folder. When Dokploy builds the application using Nixpacks, Nixpacks sees the `index.html` in the root and mistakenly triggers its **"Static Site"** provider instead of the **"Node.js"** provider. Nixpacks then packages the application with a **Caddy static server** on startup. Caddy serves the built React files perfectly, but because the Node.js/Express server is never started, any `/api/*` endpoints (such as `/api/mysql-status` or `/api/auth/session`) are captured by Caddy's single-page fallback and return the `index.html` file as a response, resulting in HTML/JSON parsing errors.

- **How we fixed this:**
  We added a custom `nixpacks.toml` file to the root of the project to tell Nixpacks to bypass auto-detection and force the **Node.js** runtime:
  ```toml
  providers = ["node"]

  [phases.install]
  cmds = [
    "rm -f package-lock.json",
    "npm install"
  ]

  [phases.build]
  cmds = [
    "npm run build"
  ]

  [start]
  cmd = "node server.cjs"
  ```

- **Action needed on your end:**
  Simply go to your **Dokploy Dashboard** under your Application settings and click **Redeploy** (do not just restart). Dokploy's Nixpacks builder will read the updated `nixpacks.toml` file, bypass the static Caddy builder, compile the application bundle correctly, and start the dynamic Express Node server on port 3000.

### ❌ Error 1: Database Connection Refused (`getaddrinfo EAI_AGAIN`) or Timeouts

- **Why this happens:**
  The Node.js container cannot find or resolve the PostgreSQL container at the specified hostname.
- **How to Solve:**
  1. Ensure both the **Application container** and the **PostgreSQL service container** are in the **same Dokploy Project** and attached to the same **Docker network** (usually `dokploy` network).
  2. Use the exact internal service host name (e.g., `sfc-portal-sfcpostdb-snp1ir`) as host, not `localhost` or `127.0.0.1`.
  3. **REDEPLOY** the Application container after saving environment variables.
